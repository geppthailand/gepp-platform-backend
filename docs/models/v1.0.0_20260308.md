## 1. Base Model

All models (unless noted) inherit from `Base` and `BaseModel`. The `BaseModel` mixin provides these common columns:

| Column | Type | Nullable | Default | Description |
|--------|------|----------|---------|-------------|
| id | BIGSERIAL (BigInteger) | NO | autoincrement | Primary key |
| is_active | BOOLEAN | NO | TRUE | Soft-active flag |
| created_date | TIMESTAMPTZ | NO | NOW() | Record creation timestamp |
| updated_date | TIMESTAMP | NO | NOW() (onupdate: NOW()) | Last update timestamp |
| deleted_date | TIMESTAMPTZ | YES | NULL | Soft-delete timestamp |

> These five columns are present on every table documented below unless explicitly stated otherwise. They are omitted from individual table definitions to avoid repetition.

## 34. Key Relationships Summary

### Organization as Central Entity
- `organizations` -> `organization_info` (1:1)
- `organizations` -> `organization_setup` (1:N, versioned)
- `organizations` -> `subscriptions` (1:N)
- `organizations` -> `user_locations` (1:N)
- `organizations` -> `organization_roles` (1:N)
- `organizations` -> `ai_audit_rule_sets` (N:1)
- `organizations` -> `organization_custom_apis` (1:N)

### User/Location Hierarchy
- `user_locations` self-referencing via `parent_user_id` (org tree)
- `user_locations` self-referencing via `parent_location_id` (location hierarchy)
- `user_subusers` many-to-many association
- `user_locations` -> `user_location_tags` (1:N)
- `user_locations` -> `user_tenants` (1:N)

### Material Hierarchy
- `material_categories` -> `main_materials` (conceptual)
- `main_materials` -> `materials` (1:N via main_material_id)
- `material_tag_groups` -> `material_tags` (via ARRAY column)
- `materials.tags` JSONB stores [(tag_group_id, tag_id)] tuples

### Transaction Flow
- `transactions` -> `transaction_records` (1:N via ARRAY column + FK)
- `transactions` -> `transaction_audits` (1:N)
- `transaction_records` -> traceability chain via `traceability` ARRAY
- `traceability_transaction_group` groups records by origin/material/tag/tenant/month
- `traceability_transport_transactions` linked to groups, self-referencing parent_id

### EPR System
- `organizations` -> `epr_organizations` (1:N via main_organization_id)
- `epr_organizations` -> `epr_brands` -> `epr_products`
- `epr_organizations` -> `epr_project`
- `epr_pro` -> `epr_pro_material_groups` -> `epr_pro_material_tag_groups` -> `epr_pro_material_tags`

### Chat System
- `chats` -> `chat_history` (1:N)
- `chats` -> `chat_meetings` (1:N)
- `chat_meetings` -> `chat_meeting_history` (1:N)
- `chat_meetings` -> `chat_meeting_participants` (1:N)
- `experts` -> `expert_knowledge_access` (1:N)
- `experts` -> `expert_configurations` (1:N)

### KM System
- `km_files` -> `km_chunks` (1:N)
- `km_chunks.embedding` Vector(1536) for semantic search
- `km_searches.query_embedding` Vector(1536) for query vectorization
- `km_files` -> `km_file_tags` (1:N)
- `km_chunks` -> `km_chunk_tags` (1:N)

### Rewards System
- `user_points` -> `user_point_transactions` (1:N)
- `claim_rules` -> `user_point_transactions` (1:N)
- `rewards` -> `reward_redemptions` (1:N)

### Logging Architecture
- `audit_logs` is the central log table
- `admin_action_logs`, `user_activity_logs`, `system_logs`, `compliance_logs`, `security_logs` all reference `audit_logs.id`
- Platform logs are independent: `platform_configuration_logs`, `feature_flag_logs`, `system_setting_logs`, `integration_config_logs`, `business_rule_logs`, `platform_metrics_logs`
