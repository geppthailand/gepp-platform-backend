## 2. Table Groups

### Group 1: Core Reference Data (Read-Heavy, Rarely Modified)
These tables contain reference/lookup data shared across the platform:
- `location_countries`, `location_regions`, `location_provinces`, `location_districts`, `location_subdistricts`
- `currencies`, `banks`, `nationalities`, `phone_number_country_codes`, `locales`
- `translations`

**Characteristics**: Pre-populated, seldom updated, no organization_id (global).

### Group 2: Material System (Hierarchical)
Three-level hierarchy:
```
MaterialCategory (e.g., "Recyclables", "Hazardous")
    └── MainMaterial (e.g., "Plastic", "Metal", "Glass")
        └── Material (e.g., "PET Bottle", "Aluminum Can")
```

- `material_categories` — Top-level classification with color coding
- `main_materials` — Sub-classification with display_order and tag_groups (ARRAY)
- `materials` — Actual items with unit_weight, calc_ghg, tags (JSONB)
- `material_tags` — Tag definitions (global or org-specific)
- `material_tag_groups` — Groups of tags (stores tag IDs as ARRAY)

**Key Design**: Materials can be `is_global=true` (shared) or `organization_id`-specific. Tags are organized by tag groups. Each material stores its tags as JSONB array of `(tag_group_id, tag_id)` tuples.

### Group 3: User & Location System (Unified Model)
The `user_locations` table is the central entity for both users and locations:
- `is_user=true`: Represents a person (has email, password, first_name, last_name)
- `is_location=true`: Represents a site (has address, coordinate, functions, population)
- Can be both simultaneously (rare)

**Hierarchy**: `parent_location_id` creates location tree. `parent_user_id` creates org chart. `organization_path` stores materialized path for fast tree queries.

**Related tables**: user_subusers (M:M), user_input_channels, user_banks, user_subscriptions, user_activities, user_devices, user_preferences, user_invitations, user_location_tags, user_tenants, integration_tokens, user_location_materials, user_reset_password_logs

### Group 4: Organization & Subscription System
- `organizations` — Tenant root entity with AI audit config
- `organization_info` — Company details (1:1)
- `organization_setup` — Versioned hierarchy structure stored as JSON (root_nodes, hub_node)
- `organization_roles` + `organization_permissions` — RBAC (M:M via organization_role_permissions)
- `subscription_plans` + `subscriptions` — Plan limits and usage tracking
- `system_roles` + `system_permissions` — Platform-level roles

### Group 5: Transaction System (Write-Heavy)
```
Transaction (batch header)
    ├── status: draft → pending → approved/rejected → completed
    ├── ai_audit_status: null → queued → approved/rejected
    ├── origin_id → UserLocation (where waste is from)
    ├── destination_ids → [UserLocation] (where waste goes)
    │
    └── TransactionRecord (line items)
        ├── material_id → Material
        ├── main_material_id → MainMaterial
        ├── category_id → MaterialCategory
        ├── origin_quantity, origin_weight_kg
        ├── destination_id → UserLocation
        └── transaction_date
```

**Audit Trail**: TransactionAudit stores each audit event (AI or human) with JSONB notes, token usage, and model version.

**Traceability**: TraceabilityTransactionGroup + TransportTransaction track material flow from origin through transport to destination.

### Group 6: EPR System (Extended Producer Responsibility)
Complete EPR lifecycle:
- `epr_organizations` — Producer/recycler entities linked to main organization
- `epr_brands` + `epr_products` — Brand and product tracking
- `epr_pros` + `epr_pro_info` — Producer Responsibility Organizations
- `epr_auditor_transaction_assignments` — Auditor assignment workflow
- `epr_notification_action_types` + `epr_notifications` — Multi-channel notifications
- `epr_payment_transaction_types` + `epr_payment_transactions` — Payment processing

### Group 7: GRI Reporting
- `gri306_1` — Waste generation and significant impacts
- `gri306_2` — Management of significant waste-related impacts
- `gri306_3` — Waste generated (spills/leaks)
- `gri306_exports` — Export file tracking

### Group 8: Rewards & Gamification
- `user_points` — Points balance and tier tracking per user per org
- `rewards` — Reward catalog with types, stock, eligibility rules
- `reward_redemptions` — Redemption requests with fulfillment tracking

### Group 9: Knowledge Management
- `km_files` — Document storage with pgvector embeddings for semantic search

### Group 10: Chat System
- `chats` — Conversation sessions with platform awareness
- Chat messages, expert configurations, meeting scheduling

### Group 11: Audit & Compliance Logging
Enterprise-grade logging with specialized log types:
- `audit_logs` — Base event log with 50+ fields
- `admin_action_logs` — Admin actions requiring approval
- `user_activity_logs` — User behavior tracking
- `system_logs` — Batch processing and system events
- `compliance_logs` — Regulatory compliance tracking
- `security_logs` — Security events and anomalies

### Group 12: Platform Operations Logging
- `platform_configuration_logs` — Config changes
- `feature_flag_logs` — Feature toggle tracking
- `system_setting_logs` — Setting modifications
- `integration_config_logs` — External service config
- `business_rule_logs` — Business rule changes
- `platform_metrics_logs` — Performance metrics snapshots

## 3. Soft Delete Pattern

All tables use `deleted_date` column:
- `NULL` = active record
- `TIMESTAMPTZ` = soft-deleted at that time

```sql
-- Standard query pattern (only active records)
SELECT * FROM user_locations WHERE organization_id = ? AND deleted_date IS NULL;

-- Include deleted (for admin/audit)
SELECT * FROM user_locations WHERE organization_id = ?;

-- Soft delete
UPDATE user_locations SET deleted_date = NOW() WHERE id = ?;

-- Restore
UPDATE user_locations SET deleted_date = NULL WHERE id = ?;
```

**Important**: The old MySQL system auto-appended `AND deleted_date IS NULL` to all fetchall queries. The new PostgreSQL system requires explicit filtering.

## 6. Index Strategy

### Primary Patterns

| Pattern | Example | Purpose |
|---------|---------|---------|
| Organization filter | `idx_transactions_org` ON transactions(organization_id) | Tenant isolation queries |
| Soft delete filter | `idx_transactions_deleted` ON transactions(deleted_date) WHERE deleted_date IS NULL | Active record queries |
| Foreign key lookup | `idx_tx_records_tx_id` ON transaction_records(created_transaction_id) | Join performance |
| Status filter | `idx_transactions_status` ON transactions(status) | Status-based queries |
| Date range | `idx_tx_records_date` ON transaction_records(transaction_date) | Report date filtering |
| Unique constraint | `idx_materials_name_org` ON materials(name_en, organization_id) | Duplicate prevention |
| GIN (JSONB) | `idx_audit_notes_gin` ON transaction_audits(audit_notes) USING GIN | JSONB queries |
| Composite | `idx_tx_org_date` ON transactions(organization_id, transaction_date) | Common query pattern |
| Migration lookup | `idx_user_locations_migration` ON user_locations(migration_id) | Data migration |

### Key Indexes per Table
- `user_locations`: organization_id, email, migration_id, parent_location_id
- `transactions`: organization_id, status, transaction_date, origin_id, migration_id
- `transaction_records`: created_transaction_id, material_id, transaction_date, migration_id
- `materials`: organization_id, category_id, main_material_id, is_global
- `audit_logs`: organization_id, action_type, created_date, actor_id
- `files`: organization_id, related_entity_type + related_entity_id

## 9. Data Migration (MySQL to PostgreSQL)

### Category ID Mapping (Old to New)
| Old MySQL ID | New PG ID | Category Name |
|-------------|-----------|---------------|
| 1 | 1 | Recyclables |
| 2 | 3 | Organic |
| 3 | 4 | General |
| 4 | 5 | Hazardous |
| 5 | 2 | Electronic |
| 6 | 9 | WTE (Waste-to-Energy) |
| 8 | 6 | Bio Hazardous |

### Platform Mapping (Old to New)
| Old MySQL | New PostgreSQL |
|-----------|---------------|
| gepp_business_web | GEPP_BUSINESS_WEB |
| gepp_epr_web | GEPP_EPR_WEB |
| gepp_reward_app | GEPP_REWARD_APP |
| gepp_backoffice | ADMIN_WEB |
| admin_web | ADMIN_WEB |
| main_backend | API |
| user_trial | WEB |
| user_paid | WEB |
| n/a | NA |

### Key Differences from Old System
1. **Unified user_locations**: Old had separate `users` and `business_units` tables, now merged into `user_locations` with `is_user`/`is_location` flags
2. **No auto-filter**: Old MySQL auto-appended `AND deleted_date IS NULL`, new PG requires explicit filter
3. **Journey dedup removed**: Old system deduplicated transaction_records by `{transaction_id}_{journey_id}`, not needed in new system
4. **Timezone handling**: Old stored as UTC-7 adjusted, new stores as UTC with TIMESTAMPTZ
5. **migration_id**: Added to user_locations, transactions, transaction_records, materials for traceability back to old MySQL IDs
