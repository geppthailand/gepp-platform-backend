## 1. Overview

GEPP (Green Environment & Plastic Platform) is a comprehensive waste management and ESG (Environmental, Social, and Governance) reporting platform. The backend is a serverless Python application deployed on AWS Lambda, backed by a PostgreSQL database managed through SQLAlchemy ORM.

This document consolidates all backend versions from v0.0.1 through v0.0.9 into a single production-ready reference for v1.0.0.

## 2. System Architecture

### 2.1 Entry Point

The main entry point is `app.py` (~1009 lines), which serves as the AWS Lambda handler. It is responsible for:

- **Request routing** across 70+ API endpoints
- **CORS handling** for cross-origin requests
- **Authentication dispatch** supporting multiple token types
- **Error handling** with standardized JSON responses

### 2.2 Deployment State Routing

The application supports two URL patterns to accommodate staged deployments:

| Pattern                            | Description             |
|------------------------------------|-------------------------|
| `/{deployment_state}/api/*`        | Stage-prefixed routes   |
| `/api/*`                           | Direct API routes       |

The handler normalizes both patterns before dispatching to service handlers.

### 2.3 Multi-Auth Support

Authentication is resolved per-request and supports the following token types:

| Token Type    | Use Case                          |
|---------------|-----------------------------------|
| JWT Bearer    | Standard user authentication      |
| Channel QR    | Public QR-based input channels    |
| Device        | IoT device authentication         |
| Integration   | Third-party system integration    |

## 4. Authentication & Authorization

### 4.1 Token Configuration

| Token Type    | Expiration  | Signing      | Description                         |
|---------------|-------------|--------------|-------------------------------------|
| Auth (Access) | 1 day       | JWT / HMAC   | Standard user access token          |
| Refresh       | 7 days      | JWT / HMAC   | Used to obtain new access tokens    |
| Device        | 1 day       | JWT / HMAC   | IoT device authentication           |
| Integration   | Custom      | Custom secret| Third-party integration access      |

### 4.2 Password Security

- Passwords are hashed with **bcrypt** using adaptive cost factor
- Password reset uses time-limited tokens sent via email (AWS Lambda email service)

### 4.3 Auth Guard

File: `libs/authGuard.py`

The auth guard middleware intercepts incoming requests and resolves the authentication context:

1. Extracts the `Authorization` header
2. Determines token type (Bearer, Channel, Device, Integration)
3. Validates and decodes the token
4. Attaches user context to the request for downstream services

### 4.4 Role-Based Access Control (RBAC)

- Organization-level roles with configurable permissions
- Preset roles provided during organization setup
- Permissions checked at the service layer before executing operations

### 5.1 Service Pattern

Each service module follows a consistent structure:

```
services/<domain>/
    __init__.py
    <domain>_handlers.py      # Request parsing, response formatting
    <domain>_service.py       # Business logic
    dto/
        __init__.py
        <domain>_requests.py  # Request DTOs / validation
        <domain>_responses.py # Response DTOs / serialization
```

- **Handlers** receive the parsed request from the router, extract parameters, call the service layer, and return formatted responses.
- **Services** contain the business logic, interact with the database via SQLAlchemy models, and raise domain-specific exceptions.
- **DTOs** define request validation schemas and response serialization formats.

## 8. AI Audit System

### 8.1 Architecture & Data Flow

The AI audit system uses **Google Gemini (gemini-2.0-flash)** via **LangChain** for image-based waste classification and audit. It operates in two modes:

1. **Custom API mode** (`POST /api/userapi/{api_path}/call`) -- External systems (e.g., BMA) push transaction data with images directly
2. **Queue mode** -- Frontend users queue transactions via `POST /api/transaction_audit/add_ai_audit_queue`, processed asynchronously by a cron job

```
┌─── Mode 1: Custom API (Synchronous) ───────────────────────┐
│  External System → POST /call → Upsert transactions        │
│  → Resolve org audit rule set → Execute AI audit            │
│  → Return results immediately                               │
└─────────────────────────────────────────────────────────────┘

┌─── Mode 2: Queue + Cron (Asynchronous) ────────────────────┐
│  Frontend "AI ตรวจสอบ" button                                │
│  → POST /add_ai_audit_queue → Set ai_audit_status='queued' │
│  → Cron (every ~30s) picks up queued transactions           │
│  → Batch AI processing → Update ai_audit_status + results  │
└─────────────────────────────────────────────────────────────┘
```

### 8.3 Audit Cron Job

Location: `crons.py` -- function `cron_process_audits()`

**Schedule:** Runs every ~30 seconds (Lambda timeout: 50 seconds max)

**Processing Algorithm:**

```
Loop while elapsed_time < 50 seconds:
  1. Check subscription limits → get allowed org_ids with remaining quota
  2. Query up to 200 transactions with ai_audit_status='queued' (ordered by id)
  3. Group transactions by organization_id
  4. For each organization:
     a. Get org-specific audit rules
     b. Prepare transaction data (flatten records + images)
     c. Call AI processing with ThreadPoolExecutor (max 25 concurrent)
     d. Update transaction.ai_audit_status (approved/rejected/failed)
     e. Store results in transaction.ai_audit_note (JSONB)
     f. Increment subscription.ai_audit_usage (successful only)
  5. Continue if < 50 seconds elapsed
```

**Key Behaviors:**
- Only updates `ai_audit_status`, NOT `transaction.status` (transaction status unchanged)
- Failed audits do NOT consume quota and retain `ai_audit_status='failed'`
- Successful audits increment `subscription.ai_audit_usage` against `subscription.ai_audit_limit`

### 8.4 Two-Phase AI Audit (BMA Rule Set)

**Step 1: Coverage Check**
- Verify transaction has required material records: `general`, `organic`, `recyclable`
- `hazardous` is optional but audited if present
- Missing required material → Reject with code `ncm` (Non-Complete Material)

**Step 2: Image Audit (Parallel via ThreadPoolExecutor)**
- For each material record with an image:
  - **2a: Visibility Check** → Gemini call with `visibility_check.yaml` prompt
  - **2b: Classification** (if visible) → Gemini call with `classify.yaml` prompt
- Images are downloaded, resized to max 384px, converted to base64 PNG

**AI Extraction Output:**

```json
{
  "bag_state": "visible|opaque|blur|empty",
  "img_quality": "ok|artificial_ui|blur",
  "is_empty_container": false,
  "main_content": "general|general_plastic|organic|recyclable|hazardous",
  "contamination_items": ["item1", "item2"],
  "contamination_pct": 0-100,
  "haz_detected": false,
  "is_heavy_liquid": false
}
```

**Decision Rules by Claimed Material Type:**

| Claimed Type | Condition | Decision |
|---|---|---|
| General (94) | Pure recyclable/organic detected | Reject (WC) |
| General (94) | Mixed/general detected | Approve (CC) |
| Organic (77) | Wrong type detected | Reject (WC) |
| Organic (77) | Contamination >20% | Reject (WC) |
| Organic (77) | Contamination 0-20% | Approve (LC) |

### 8.5 Transaction Fields Updated by AI Audit

| Field | Type | Description |
|-------|------|-------------|
| `ai_audit_status` | Enum | `null` → `queued` → `approved`/`rejected`/`failed` |
| `ai_audit_note` | JSONB | `{ status, audits: {...}, v: [{id, m}] }` -- detailed per-material results and violations |
| `ai_audit_date` | DateTime | Timestamp when AI audit was performed |
| `audit_tokens` | JSONB | `{ input_tokens, output_tokens, total_tokens, thinking_tokens }` |
| `reject_triggers` | Array | Rule IDs that triggered rejection |
| `warning_triggers` | Array | Rule IDs that triggered warnings |

### 8.6 Prompt Templates

Prompts are stored as **YAML files** in `prompts/ai_audit_v1/bma/`:

| File | Purpose |
|------|---------|
| `visibility_check.yaml` | Check if waste content is visible in image |
| `classify.yaml` | Classify material type and detect contamination |

### 8.7 Models

| Model | Table | Description |
|-------|-------|-------------|
| `AiAuditRuleSet` | `ai_audit_rule_sets` | Audit configurations with `function_name` pointing to Python module (e.g., `bma_audit_rule_set`) |
| `AiAuditResponsePattern` | `ai_audit_response_patterns` | Customizable message templates per org/material with condition codes and `{{code}}`, `{{detect_type}}`, `{{claimed_type}}`, `{{warning_items}}` placeholders |

### 8.8 Quota Management

| Level | Limit Field | Usage Field | Description |
|-------|-------------|-------------|-------------|
| Subscription | `ai_audit_limit` | `ai_audit_usage` | Max transactions per subscription period |
| Custom API (OrganizationCustomApi) | `api_call_quota` / `process_quota` | `api_call_used` / `process_used` | Per-org API call and image processing limits |

### 8.9 Token Usage Tracking

- Each Gemini call tracks usage via `response.usage_metadata`
- Two-phase audit accumulates tokens from both visibility + classification calls
- Cron uses `extraction_mode='minimal'` to reduce token consumption
- Model: `gemini-2.0-flash` (or `gemini-2.5-flash-lite` for cost optimization)

## 9. File Storage

### 9.1 AWS S3

All file uploads are managed through AWS S3 with presigned URLs:

| Operation       | Flow                                                      |
|-----------------|------------------------------------------------------------|
| **Upload**      | Client requests presigned URL -> uploads directly to S3   |
| **View**        | Client requests presigned view URL -> accesses file via URL|

### 9.2 Supported File Types

- Transaction images (waste evidence photos)
- Profile images (user avatars)
- Report documents (PDF exports)

### 9.3 File Model

The `File` model tracks metadata for each uploaded file:

| Field          | Description                           |
|----------------|---------------------------------------|
| `id`           | Unique file identifier                |
| `file_name`    | Original filename                     |
| `file_type`    | MIME type                             |
| `s3_key`       | S3 object key                         |
| `status`       | Upload status (pending, completed)    |
| `observations` | AI-generated observations (if audited)|
