🔍 หมวดหมู่การตรวจจับความผิดปกติ

Category 1: ความครบถ้วน (Completeness Validation)

✅ ตรวจสอบข้อมูลที่จำเป็นต้องมี

🚨 1.1 Required Fields

  • Location: ต้องเลือก origin_id (ไม่สามารถเป็นค่าว่างได้)
  • Material Records: ต้องมีอย่างน้อย 1 รายการ
  • Date: transaction_date ต้องไม่เป็นค่าว่าง
  • Material Type: material_id ต้องถูกเลือก
  • Weight/Quantity: origin_weight_kg > 0
❌ Error Message: "กรุณากรอกข้อมูลให้ครบถ้วน: [รายการที่ขาด]"

⚠️ 1.2 Recommended Fields

  • Price: origin_price_per_unit แนะนำให้กรอก (บางวัสดุอาจไม่มีราคา)
  • Notes: หากมี anomaly ควรมี notes อธิบาย
  • Images: สำหรับรายการมูลค่าสูง (>5,000 บาท) ควรมีภาพประกอบ
⚠️ Warning Message: "แนะนำให้กรอกข้อมูลเพิ่มเติม: [รายการที่แนะนำ]"

Category 2: ความถูกต้อง (Data Correctness)

✅ ตรวจสอบความถูกต้องของข้อมูล

🔢 2.1 Numeric Constraints

  • Weight: origin_weight_kg > 0 และ < 10,000 kg
  • Price: origin_price_per_unit >= 0 และ < 1,000,000 บาท/หน่วย
  • Total Amount: total_amount = weight × price (ต้องตรงกัน)
  • Negative Values: ไม่อนุญาตให้มีค่าติดลบ
❌ Error Examples:
• "น้ำหนัก -5 kg: ค่าติดลบไม่ถูกต้อง"
• "ราคา 1,500,000 บาท/kg: เกินขีดจำกัด"
• "ราคารวมไม่ตรงกับการคำนวณ (50 kg × 10 บาท ≠ 600 บาท)"

📅 2.2 Date/Time Validation

  • Date Range: transaction_date ต้องไม่เป็นอนาคต
  • Past Limit: ไม่ควรย้อนหลังเกิน 90 วัน (ปรับได้ตาม policy)
  • Duplicate Detection: ไม่ควรมี 2 transactions จากสถานที่เดียวกันใน 1 ชั่วโมง
  • Working Hours: แจ้งเตือนถ้ากรอกนอกเวลาทำการ (22:00-06:00)
⚠️ Warning Examples:
• "วันที่ 15 ต.ค. 2566: ย้อนหลัง 120 วัน ควรตรวจสอบ"
• "เวลา 02:35: นอกเวลาทำการ อาจไม่ปกติ"
• "พบรายการซ้ำจากสถานที่เดียวกัน 30 นาทีที่แล้ว"

🎯 2.3 Material-Specific Rules

  • Material Compatibility: ตรวจสอบว่าวัสดุสามารถอยู่ที่ location นี้ได้หรือไม่
  • Hazardous Materials: วัสดุอันตราย (material.hazardous_level > 0) ต้องมี special approval
  • Price Range: ราคาต้องอยู่ในช่วง min-max ของวัสดุนั้น (จาก master data)
  • Unit Mismatch: หน่วยต้องตรงกับที่วัสดุกำหนด (kg, piece, ton)
❌ Error Examples:
• "วัสดุอันตราย (แบตเตอรี่) ต้องได้รับอนุมัติพิเศษ"
• "ราคา PET Plastic 150 บาท/kg: สูงกว่าช่วงปกติ (5-50 บาท/kg)"
• "วัสดุนี้ใช้หน่วย 'piece' ไม่ใช่ 'kg'"

Category 3: ความผิดปกติทางสถิติ (Statistical Anomaly)

📊 เปรียบเทียบกับข้อมูลในอดีต

📈 3.1 Transaction-Level Anomaly (รายการเดียว)

  • Weight Outlier: น้ำหนักต่างจากค่าเฉลี่ยมาก (Z-score > 3)
  • Price Outlier: ราคาต่างจากราคาตลาดมาก
  • Record Count: จำนวน records ต่อ transaction แปลกผิดปกติ
  • Material Mix: ส่วนผสมของวัสดุผิดปกติ
# Example: Weight Anomaly Detection
SELECT
    AVG(weight_kg) as mean_weight,
    STDDEV(weight_kg) as std_weight
FROM transactions
WHERE organization_id = ?
  AND material_id = ?
  AND transaction_date >= NOW() - INTERVAL '60 days';

# Calculate Z-score
z_score = (current_weight - mean_weight) / std_weight

# Flag if Z-score > 3
IF z_score > 3:
    WARNING: "น้ำหนัก {current_weight} kg สูงผิดปกติ (เฉลี่ย {mean_weight} kg, Z={z_score:.2f})"
⚠️ Warning Examples:
• "น้ำหนัก PET Plastic 250 kg: สูงกว่าค่าเฉลี่ย 50 kg (Z=4.5) 🚨"
• "ราคา 8 บาท/kg: ต่ำกว่าช่วงปกติ 15-25 บาท/kg"
• "มีรายการย่อย 25 รายการ: มากกว่าปกติ (เฉลี่ย 3-5 รายการ)"

📆 3.2 Batch-Level Anomaly (ภาพรวมรายวัน/สัปดาห์/เดือน)

  • Daily Volume Spike: ปริมาณรวมต่อวันเพิ่มขึ้นกะทันหัน (>200% จากเฉลี่ย)
  • Frequency Anomaly: จำนวนรายการต่อวันผิดปกติ (เช่น 50 รายการ/วัน ทั้งที่ปกติ 5-10)
  • Temporal Pattern: เวลากรอกข้อมูลแปลกผิดปกติ (เช่น กรอกข้อมูล 30 วันในคราวเดียว)
  • Location Pattern: สถานที่เดียวกันมีรายการมากผิดปกติ
# Example: Daily Volume Spike Detection
WITH daily_stats AS (
    SELECT
        DATE(transaction_date) as txn_date,
        SUM(weight_kg) as daily_weight
    FROM transactions
    WHERE organization_id = ?
      AND transaction_date >= NOW() - INTERVAL '30 days'
    GROUP BY DATE(transaction_date)
)
SELECT
    AVG(daily_weight) as avg_daily,
    STDDEV(daily_weight) as std_daily
FROM daily_stats;

# Check today's total
today_weight = SUM(weight_kg WHERE DATE(transaction_date) = TODAY)
spike_ratio = today_weight / avg_daily

IF spike_ratio > 2:
    WARNING: "ปริมาณวันนี้ {today_weight} kg: สูงกว่าเฉลี่ย {avg_daily} kg ({spike_ratio:.1f}x) 🚨"
⚠️ Warning Examples:
• "ปริมาณวันนี้ 1,200 kg: สูงกว่าค่าเฉลี่ย 300 kg (4x) 🚨"
• "มีรายการ 45 รายการวันนี้: มากกว่าปกติ 8-12 รายการ"
• "ผู้ใช้ A กรอกข้อมูล 20 รายการในรอบ 5 นาที: อาจเป็น bulk import"
• "สถานที่ 'ชั้น 3 ห้อง 301' มี 8 รายการวันนี้: ผิดปกติ (ปกติ 1-2 รายการ/สัปดาห์)"

🔄 3.3 Trend & Pattern Detection

  • Sudden Change: พฤติกรรมเปลี่ยนแปลงกะทันหัน
  • Gradual Increase: ปริมาณเพิ่มขึ้นทีละน้อยแต่ต่อเนื่อง
  • Seasonality Break: ผิดจาก pattern ตามฤดูกาล
  • Gap Detection: ช่วงเงียบผิดปกติ (เช่น ไม่มีข้อมูล 7 วัน)
💡 Analysis Methods:
• Moving Average (7-day, 30-day): เฉลี่ยเคลื่อนที่เพื่อหา trend
• Week-over-Week Comparison: เปรียบเทียบสัปดาห์นี้ vs สัปดาห์ก่อน
• Day-of-Week Pattern: เปรียบเทียบจันทร์ vs จันทร์ ไม่ใช่ จันทร์ vs อาทิตย์

Category 4: การตรวจจับการฉ้อโกง (Fraud Detection)

🚨 Suspicious Patterns

💰 4.1 Financial Fraud Indicators

  • Round Numbers: ราคาเป็นเลขกลมๆ มากเกินไป (เช่น 1000, 5000, 10000)
  • Consistent Pricing: ราคาเท่ากันหมดทุกครั้ง (ไม่มีความแปรปรวน)
  • Just Below Threshold: มูลค่าต่ำกว่าเกณฑ์นิดหน่อย (เช่น threshold 10,000 → input 9,999)
  • Price Manipulation: ปรับราคาให้ได้มูลค่าที่ต้องการ
  • Fake Transactions: รายการที่มี pattern เหมือนกันมาก
🚨 Red Flags:
• "ราคา 10,000 บาท (เลขกลม) ปรากฏ 5 ครั้งติดต่อกัน"
• "มูลค่า 9,950 บาท: ต่ำกว่า threshold 10,000 นิดหน่อย 🚨"
• "ผู้ใช้ B กรอกรายการ 15 รายการ มูลค่ารวม 149,500 บาท ในวันเดียว"
• "ราคา PET ของ User A = 50 บาท/kg ทุกครั้ง (15 ครั้ง) ไม่มีความผันแปร"

👤 4.2 User Behavior Anomaly

  • Bulk Entry: กรอกข้อมูลจำนวนมากในเวลาสั้น
  • Off-Hours Activity: กรอกข้อมูลนอกเวลาทำการบ่อย
  • Location Hopping: สลับ location บ่อยผิดปกติ
  • Copy-Paste Pattern: ข้อมูลเหมือนกันมาก (น้ำหนัก, ราคา)
  • New User Spike: ผู้ใช้ใหม่กรอกข้อมูลปริมาณมาก
⚠️ Warning Examples:
• "User C กรอก 25 รายการในรอบ 3 นาที (เฉลี่ย 7.2 วินาที/รายการ)"
• "User D กรอกข้อมูลเวลา 23:45-01:30 (3 ครั้งในสัปดาห์นี้)"
• "น้ำหนัก 45.50 kg ปรากฏ 8 ครั้ง: อาจ copy-paste"

🔗 4.3 Relationship Anomaly

  • Same Origin-Destination: ใช้คู่เดิมซ้ำๆ ผิดปกติ
  • Material Concentration: วัสดุ 1 ชนิดเป็นส่วนใหญ่ (>80%)
  • Cross-Organization: มี pattern เหมือนกันระหว่าง org (อาจเป็น organized fraud)
  • Coordinated Activity: หลายผู้ใช้กรอกข้อมูลพร้อมกัน
💡 Advanced Detection:
• Graph Analysis: วิเคราะห์ network ระหว่าง user-location-material
• Clustering: จัดกลุ่มพฤติกรรมที่น่าสงสัย
• Correlation: หาความสัมพันธ์ที่ไม่ปกติ

Frontend Implementation (React)

// Example: Real-time Weight Validation
const WeightInput = ({ materialId, locationId, value, onChange }) => {
  const [anomalyScore, setAnomalyScore] = useState(null);
  const [baseline, setBaseline] = useState(null);

  useEffect(() => {
    // Fetch baseline on mount
    fetchBaseline(materialId, locationId).then(setBaseline);
  }, [materialId, locationId]);

  useEffect(() => {
    // Calculate anomaly score on value change
    if (baseline && value) {
      const zScore = (value - baseline.mean) / baseline.stdDev;
      setAnomalyScore({
        zScore,
        severity: zScore > 3 ? 'error' : zScore > 2 ? 'warning' : 'normal'
      });
    }
  }, [value, baseline]);

  return (
    <div className={`weight-input ${anomalyScore?.severity}`}>
      <input
        type="number"
        value={value}
        onChange={(e) => onChange(e.target.value)}
      />
      {baseline && (
        <span className="baseline-hint">
          เฉลี่ย: {baseline.mean.toFixed(2)} kg
        </span>
      )}
      {anomalyScore?.severity === 'error' && (
        <div className="error-message">
          ⚠️ น้ำหนักสูงผิดปกติ (Z={anomalyScore.zScore.toFixed(2)})
        </div>
      )}
    </div>
  );
};

Database Schema Extensions

-- Table: anomaly_baselines (เก็บ baseline สำหรับการเปรียบเทียบ)
CREATE TABLE anomaly_baselines (
    id BIGSERIAL PRIMARY KEY,
    organization_id BIGINT NOT NULL,
    material_id BIGINT,
    origin_id BIGINT,
    window_days INT NOT NULL, -- 30/60/90

    -- Statistical metrics
    metric_name VARCHAR(50) NOT NULL, -- weight_kg, price, frequency
    mean_value DECIMAL(15,4),
    median_value DECIMAL(15,4),
    std_dev DECIMAL(15,4),
    min_value DECIMAL(15,4),
    max_value DECIMAL(15,4),
    q1_value DECIMAL(15,4),  -- 25th percentile
    q3_value DECIMAL(15,4),  -- 75th percentile
    sample_count INT,

    calculated_date TIMESTAMP,
    created_date TIMESTAMP DEFAULT NOW(),
    UNIQUE(organization_id, material_id, origin_id, window_days, metric_name)
);

-- Table: transaction_validations (เก็บผล validation ของแต่ละ transaction)
CREATE TABLE transaction_validations (
    id BIGSERIAL PRIMARY KEY,
    transaction_id BIGINT NOT NULL REFERENCES transactions(id),

    -- Overall scores
    validity_score DECIMAL(5,2) NOT NULL, -- 0-100
    anomaly_score DECIMAL(5,2) NOT NULL, -- 0-100
    fraud_score DECIMAL(5,2) NOT NULL, -- 0-100

    -- Issues found
    blocker_count INT DEFAULT 0,
    warning_count INT DEFAULT 0,
    info_count INT DEFAULT 0,

    -- Detailed issues (JSONB)
    issues JSONB NOT NULL, -- Array of issue objects

    -- Batch context
    batch_analysis JSONB, -- Daily/weekly stats

    -- Status
    validation_status VARCHAR(20) DEFAULT 'pending', -- pending, approved, flagged, rejected
    reviewed_by BIGINT REFERENCES user_locations(id),
    reviewed_date TIMESTAMP,
    review_notes TEXT,

    created_date TIMESTAMP DEFAULT NOW(),
    INDEX idx_transaction_id (transaction_id),
    INDEX idx_validity_score (validity_score),
    INDEX idx_anomaly_score (anomaly_score),
    INDEX idx_fraud_score (fraud_score),
    INDEX idx_validation_status (validation_status)
);

-- Table: validation_rules (เก็บ configuration ของกฎการตรวจสอบ)
CREATE TABLE validation_rules (
    id BIGSERIAL PRIMARY KEY,
    organization_id BIGINT NOT NULL,
    rule_code VARCHAR(50) NOT NULL, -- weight_outlier, price_outlier, etc.
    rule_type VARCHAR(20) NOT NULL, -- completeness, correctness, anomaly, fraud
    severity VARCHAR(20) NOT NULL, -- blocker, warning, info

    -- Rule configuration (JSONB)
    config JSONB NOT NULL, -- Thresholds, parameters

    is_enabled BOOLEAN DEFAULT true,
    created_date TIMESTAMP DEFAULT NOW(),
    updated_date TIMESTAMP,
    UNIQUE(organization_id, rule_code)

Services Architecture

1️⃣ Baseline Calculator Service
หน้าที่: คำนวณ baselines ทุกคืน
  • • Cron job ทำงาน 02:00 ทุกวัน
  • • คำนวณ mean, std dev, percentiles
  • • Rolling windows: 30/60/90 days
  • • Group by: org, material, location
2️⃣ Real-time Validator Service
หน้าที่: ตรวจสอบ transaction แบบ real-time
  • • ถูกเรียกจาก frontend (ก่อน submit)
  • • Run ทุก validation rules
  • • Return issues + suggestions
  • • Response time < 200ms
3️⃣ Batch Analyzer Service
หน้าที่: วิเคราะห์ภาพรวมรายวัน/สัปดาห์
  • • คำนวณ daily/weekly aggregates
  • • Detect batch-level anomalies
  • • Generate summary reports
  • • Alert admins if needed
4️⃣ Fraud Detection Service
หน้าที่: ตรวจจับ fraud patterns
  • • Run fraud detection algorithms
  • • Graph analysis (user-location)
  • • Clustering suspicious behavior
  • • Flag high-risk transactions
