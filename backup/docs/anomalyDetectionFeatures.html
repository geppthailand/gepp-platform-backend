Transaction & TransactionRecord Schema

📦 Transaction (รายการซื้อขาย/ส่งมอบขยะแต่ละครั้ง)
ข้อมูลตัวเลขที่สำคัญ:
  • • weight_kg (DECIMAL 15,4): น้ำหนักรวมทั้งหมด (kg) ⭐ Primary Metric
  • • total_amount (DECIMAL 15,4): มูลค่ารวมทั้งหมด (บาท)
  • • hazardous_level (0-5): ระดับอันตราย
  • • transaction_records (ARRAY): จำนวนรายการย่อย
ข้อมูลบริบท:
  • • organization_id: องค์กร
  • • origin_id: สถานที่ต้นทาง
  • • transaction_date: วันที่ทำรายการ
  • • transaction_method: วิธีการ (origin/transport/transform/qr_input/scale_input)
  • • status: สถานะ (draft/pending/approved/rejected/completed)
  • • images (JSONB): รูปภาพประกอบ
  • • notes: บันทึกเพิ่มเติม
📋 TransactionRecord (รายการย่อยแต่ละวัสดุ)
ข้อมูลตัวเลขที่สำคัญ:
  • • origin_weight_kg (DECIMAL 15,4): น้ำหนัก (kg) ⭐ Primary Metric
  • • origin_quantity (DECIMAL 15,4): ปริมาณ
  • • origin_price_per_unit (DECIMAL 15,4): ราคาต่อหน่วย
  • • total_amount (DECIMAL 15,4): มูลค่ารวม
  • • hazardous_level (0-5): ระดับอันตราย
ข้อมูลบริบท:
  • • material_id: วัสดุเฉพาะเจาะจง
  • • main_material_id: หมวดหมู่วัสดุหลัก
  • • category_id: หมวดหมู่ใหญ่
  • • destination_id: สถานที่ปลายทาง
  • • unit: หน่วย (kg, ton, piece, etc.)

🎯 แนวทางการตรวจจับความผิดปกติ

Level 1: Statistical Anomaly Detection (พื้นฐาน - ใช้ทุกรายการ)

📈 Z-Score & IQR Analysis

หลักการ: เปรียบเทียบข้อมูลปัจจุบันกับค่าเฉลี่ย (mean) และส่วนเบี่ยงเบนมาตรฐาน (std dev) ของข้อมูลในอดีต

Metrics ที่ตรวจสอบ:

  • Weight Anomaly: น้ำหนักแต่ละรายการ vs ค่าเฉลี่ยของวัสดุชนิดนั้น
  • Price Anomaly: ราคาต่อหน่วย vs ราคาตลาดปกติ
  • Volume Anomaly: จำนวน records ต่อ transaction
  • Frequency Anomaly: จำนวนรายการต่อวัน/สัปดาห์
  • Time Pattern Anomaly: รายการนอกเวลาทำการ/วันหยุด

Thresholds:

  • ปกติ: Z-score < 2 (within 95% confidence)
  • น่าสงสัย: 2 ≤ Z-score < 3
  • ผิดปกติมาก: Z-score ≥ 3

ตัวอย่างการคำนวณ:

z_score = (current_weight - mean_weight) / std_dev_weight

# Example:
# - Mean weight for PET plastic: 50 kg
# - Std dev: 10 kg
# - Current transaction: 85 kg
# - Z-score = (85 - 50) / 10 = 3.5 (ผิดปกติมาก!)
ข้อดี:
  • ✓ รวดเร็วมาก (<50ms per transaction)
  • ✓ ไม่มีค่าใช้จ่าย API
  • ✓ ทำงานแบบ real-time ได้
  • ✓ อธิบายได้ง่าย (explainable)
ข้อจำกัด:
  • ⚠️ ต้องการข้อมูลในอดีตอย่างน้อย 30-60 รายการ
  • ⚠️ อาจมี false positive ในช่วงเปลี่ยนฤดูกาล
  • ⚠️ ไม่เข้าใจ context ที่ซับซ้อน

Level 2: Time Series Analysis (กลาง - ใช้ทุกรายการ)

📉 Trend & Seasonality Detection

หลักการ: วิเคราะห์รูปแบบเชิงเวลา (temporal patterns) เพื่อหา trends และ seasonality

Patterns ที่ตรวจจับ:

  • Sudden Spike: ปริมาณเพิ่มขึ้นกะทันหัน (เช่น 5x จากสัปดาห์ก่อน)
  • Sudden Drop: ปริมาณลดลงกะทันหัน (เช่น drop 80%)
  • Unusual Timing: รายการในเวลาผิดปกติ (เช่น 3 AM, วันหยุด)
  • Frequency Change: เปลี่ยนจาก daily → hourly หรือ weekly → daily
  • Gap Detection: ช่วงเงียบผิดปกติ (เช่น ไม่มีรายการ 7 วัน ทั้งที่ปกติทุกวัน)

Methods:

  • Moving Average: เฉลี่ยเคลื่อนที่ 7/30 วัน
  • Exponential Smoothing: ให้น้ำหนักข้อมูลล่าสุดมากกว่า
  • Day-of-Week Pattern: เปรียบเทียบจันทร์ vs จันทร์ ไม่ใช่ จันทร์ vs อาทิตย์
ข้อดี:
  • ✓ จับ sudden changes ได้ดีกว่า Z-score
  • ✓ เข้าใจ seasonality (เช่น สิ้นเดือนมีขยะเยอะ)
  • ✓ ยังคงรวดเร็วและไม่มีค่าใช้จ่าย

Level 3: Rule-Based Detection (กลาง - ใช้ทุกรายการ)

⚖️ Business Rules & Constraints

หลักการ: ตรวจสอบตามกฎธุรกิจที่กำหนดไว้ล่วงหน้า

Rule Categories:

1. Physical Constraints:
  • • Weight must be positive (weight_kg > 0)
  • • Weight must match material density expectations
  • • Volume must be reasonable for container type
  • • Hazardous level must match material type
2. Business Constraints:
  • • Price must be within market range (min-max bounds)
  • • Organization daily limits (max weight per day)
  • • Location capacity constraints
  • • Hazardous material requires special approval
3. Relationship Constraints:
  • • Same origin shouldn't have drastically different prices
  • • Same material shouldn't have wild weight variations
  • • Destination must be logical for material type
ข้อดี:
  • ✓ Explainable 100% (บอกได้ชัดว่าผิดกฎไหน)
  • ✓ False positive ต่ำมาก
  • ✓ Customizable ตาม business needs

Level 4: LLM-Powered Deep Analysis (ขั้นสูง - ใช้เฉพาะรายการที่ flag)

🤖 AI-Powered Contextual Understanding

หลักการ: ใช้ LLM (Gemini 2.0 Flash) เพื่อวิเคราะห์เชิงลึกเฉพาะรายการที่ผ่าน Level 1-3 แล้วยังสงสัย

🎯 เมื่อไหร่ใช้ LLM:

  • ✓ Level 1-3 flag เป็น "suspicious"
  • ✓ Multiple anomaly types detected พร้อมกัน
  • ✓ High-value transactions (>100,000 บาท)
  • ✓ Hazardous materials involved
  • ✓ Cross-organization transactions

What LLM Analyzes:

  • Images: ดูรูปภาพว่าสอดคล้องกับข้อมูลที่บันทึกหรือไม่
  • Notes: อ่าน notes หาคำอธิบาย/เหตุผลที่เป็นไปได้
  • Historical Context: ดูประวัติ 30-90 วันย้อนหลัง
  • Relationships: วิเคราะห์ความสัมพันธ์ระหว่าง origin-destination-material
  • Multi-factor Correlation: หาความสัมพันธ์ซับซ้อนที่ statistical model จับไม่ได้

Prompt Strategy (เพื่อลด token):

You are a waste transaction auditor. Analyze this flagged transaction:

**Current Transaction:**
- ID: {transaction_id}
- Weight: {weight_kg} kg (vs avg: {avg_weight} kg)
- Price: {total_amount} THB
- Material: {material_name}
- Origin: {origin_name}
- Date: {transaction_date}
- Statistical Flags: {anomaly_flags}

**Recent History (last 30 days):**
{compact_history}

**Images:** {image_urls}
**Notes:** {notes}

**Question:** Is this transaction legitimately unusual or potentially fraudulent?

Respond in JSON:
{
  "is_suspicious": boolean,
  "confidence": 0-100,
  "reason": "brief explanation in Thai",
  "recommended_action": "approve|review|reject"
}
ข้อดี:
  • ✓ Contextual understanding (เข้าใจบริบท)
  • ✓ Image analysis (วิเคราะห์ภาพได้)
  • ✓ Natural language reasoning (อธิบายเหตุผลได้)
  • ✓ Catches complex fraud patterns
ข้อจำกัด & Cost Management:
  • ⚠️ Cost: ~฿0.30-0.50 per transaction
  • ⚠️ Latency: 2-5 seconds
  • ⚠️ Token limits: Keep prompt under 4,000 tokens
  • ⚠️ Rate limits: Max 60 requests/minute

💡 Strategy: ใช้เฉพาะ 5-10% ของรายการที่ผ่าน statistical filter แล้ว → รักษา cost ให้ต่ำ

Component Overview

📊 Baseline Calculator Service
หน้าที่: คำนวณ baselines สำหรับแต่ละ organization/material/origin
  • • Daily cron job (ทำทุกคืน)
  • • Calculate: mean, median, std dev, min, max
  • • Rolling windows: 30/60/90 days
  • • Store in anomaly_baselines table
🔍 Anomaly Detector Service
หน้าที่: ตรวจสอบ transaction ใหม่เมื่อสร้าง
  • • Called on transaction create/update
  • • Run Level 1-3 checks (<100ms)
  • • Calculate anomaly_score (0-100)
  • • Flag if score > threshold
🤖 LLM Analyzer Service
หน้าที่: Deep analysis ด้วย LLM
  • • Triggered for high-risk transactions
  • • Fetch transaction + history + images
  • • Call Gemini API with structured prompt
  • • Parse JSON response
📈 Dashboard & Reports
หน้าที่: UI สำหรับดู/จัดการ anomalies
  • • List flagged transactions
  • • Show anomaly scores & reasons
  • • Approve/Reject actions
  • • False positive feedback

Database Schema Changes

-- New table: anomaly_baselines
CREATE TABLE anomaly_baselines (
    id BIGSERIAL PRIMARY KEY,
    organization_id BIGINT NOT NULL,
    material_id BIGINT,
    origin_id BIGINT,
    window_days INT NOT NULL, -- 30/60/90
    metric_name VARCHAR(50) NOT NULL, -- weight_kg, price, frequency
    mean_value DECIMAL(15,4),
    median_value DECIMAL(15,4),
    std_dev DECIMAL(15,4),
    min_value DECIMAL(15,4),
    max_value DECIMAL(15,4),
    sample_count INT,
    calculated_date TIMESTAMP,
    created_date TIMESTAMP DEFAULT NOW(),
    UNIQUE(organization_id, material_id, origin_id, window_days, metric_name)
);

-- New table: transaction_anomalies
CREATE TABLE transaction_anomalies (
    id BIGSERIAL PRIMARY KEY,
    transaction_id BIGINT NOT NULL REFERENCES transactions(id),
    anomaly_score DECIMAL(5,2) NOT NULL, -- 0-100
    anomaly_types JSONB NOT NULL, -- ["weight_spike", "price_outlier"]
    statistical_analysis JSONB, -- z-scores, thresholds, etc.
    llm_analysis JSONB, -- LLM response if analyzed
    status VARCHAR(20) DEFAULT 'pending', -- pending, approved, rejected, false_positive
    reviewed_by BIGINT REFERENCES user_locations(id),
    reviewed_date TIMESTAMP,
    created_date TIMESTAMP DEFAULT NOW(),
    INDEX idx_transaction_id (transaction_id),
    INDEX idx_status (status),
    INDEX idx_anomaly_score (anomaly_score)
);

-- Add fields to transactions table
ALTER TABLE transactions ADD COLUMN anomaly_score DECIMAL(5,2);
ALTER TABLE transactions ADD COLUMN anomaly_status VARCHAR(20); -- clean, flagged, under_review, confirmed_fraud
ALTER TABLE transactions ADD COLUMN anomaly_checked_date TIMESTAMP;
